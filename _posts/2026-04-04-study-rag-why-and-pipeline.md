# RAG 완전 가이드 1: 필요성과 기본 구조

> **한줄 정의**
> RAG는 LLM이 답변하기 전에 외부 지식을 검색하고, 그 결과를 context로 붙여 답변하게 만드는 구조다.

## 왜 필요한가

LLM은 자체 기억만으로 답변한다. 그래서 다음 문제가 생긴다.

| 문제 | 설명 | RAG의 역할 |
| --- | --- | --- |
| 할루시네이션 | 근거 없이 그럴듯한 답을 생성 | 답변 전에 근거 문서를 제공 |
| 지식 단절 | 학습 이후 정보나 사내 문서를 모름 | 최신 문서와 내부 데이터를 검색 |
| 도메인 지식 부족 | 전문 업무 문맥을 모름 | 업무 문서를 context로 주입 |
| 출처 불투명 | 답변 근거를 추적하기 어려움 | citation과 문서 ID를 함께 남김 |

RAG는 모델을 다시 학습시키는 방식이 아니다. 지식을 모델 안에 넣는 대신, 답변 시점에 필요한 지식을 찾아 붙이는 방식이다.

## RAG의 비유

일반 LLM은 클로즈드북 시험에 가깝다. 모델 내부 기억만으로 답한다.

RAG는 오픈북 시험에 가깝다. 질문을 받은 뒤 관련 자료를 찾아 읽고 답한다.

```text
User Query
  -> Retrieve
  -> Augment
  -> Generate
  -> Answer with Evidence
```

## 핵심 파이프라인

![RAG core pipeline](/assets/images/study/diagrams/study-rag-as-product-pipeline.svg){: width="100%"}

| 단계 | 역할 | 실패하면 생기는 문제 |
| --- | --- | --- |
| Query | 사용자의 질문 | 질문 의도가 모호하면 검색도 흔들림 |
| Retrieve | 관련 문서 검색 | 필요한 문서가 context에 들어오지 않음 |
| Augment | 검색 결과를 프롬프트에 삽입 | 너무 많은 노이즈가 들어가거나 근거가 빠짐 |
| Generate | LLM이 답변 생성 | 근거를 무시하거나 없는 내용을 생성 |

RAG 품질은 LLM 하나로 결정되지 않는다. 검색, 청킹, 메타데이터, 재순위화, 평가가 함께 맞아야 한다.

## 임베딩

임베딩은 텍스트를 숫자 벡터로 바꾸는 과정이다.

```text
"고양이" -> [0.12, -0.45, 0.78, ...]
"cat"    -> [0.10, -0.40, 0.75, ...]
```

의미가 비슷한 문장은 벡터 공간에서 가까워진다. 그래서 키워드가 완전히 같지 않아도 비슷한 의미의 문서를 찾을 수 있다.

## 벡터 데이터베이스

벡터 DB는 임베딩 벡터를 저장하고 유사도 검색을 수행한다.

| 구성 | 역할 |
| --- | --- |
| Vector | 문서 chunk의 의미 표현 |
| Metadata | 문서 날짜, 권한, 섹션, 출처 ID |
| Index | HNSW, IVF 같은 빠른 검색 구조 |
| Filter | 사용자 권한이나 문서 유형 조건 |

벡터 DB만 붙이면 RAG가 완성되는 것이 아니다. 권한 필터, 문서 버전, chunk 단위, 검색 결과 평가가 같이 있어야 한다.

## 시맨틱 검색

시맨틱 검색은 키워드가 아니라 의미 유사성으로 찾는다.

예를 들어 `"연봉 인상 방법"`이라는 질문은 `"급여 협상 전략"` 문서와 의미상 가까울 수 있다.

| 질문 | 검색될 수 있는 문서 | 이유 |
| --- | --- | --- |
| 연봉 인상 방법 | 급여 협상 전략 | 표현은 다르지만 의미가 비슷함 |
| 휴가 어떻게 써 | 연차 휴가 사용 절차 | 구어체 질문을 업무 문서와 연결 |
| 로그인 오류 | 인증 실패 처리 가이드 | 증상과 원인 문서 연결 |

## RAG의 출발점

RAG 연구의 출발점은 검색기와 생성기를 결합하는 구조다. 원본 학습 노트 기준으로 RAG-Sequence와 RAG-Token은 검색된 문서를 생성 과정에 어떻게 반영할지에 대한 초기 변형으로 정리된다.

현재 실무 RAG는 이 기본 구조에서 훨씬 확장됐다.

```text
Naive RAG
  -> Advanced RAG
  -> Modular RAG
  -> Agentic RAG
  -> Context Engineering
```

## 내 기준

RAG는 "LLM에 문서를 넣는 기술"이 아니라 "질문에 맞는 근거를 안정적으로 context에 올리는 시스템"이다.

RAG를 설계할 때 첫 질문은 모델이 아니다.

```text
답이 원본에 있는가
  -> 그 원본이 검색 가능한가
  -> 검색된 context가 충분한가
  -> 답변이 context에 충실한가
  -> 운영 중 품질을 측정할 수 있는가
```
